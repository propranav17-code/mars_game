# Project Rules

This document defines **how Project Mars is developed**.

These rules are intended to keep the project focused, maintainable, and true to its original vision.

Every contributor should read this document before making changes to the project.

---

# Rule 1 — The Concept Bible Is the Highest Authority

The Concept Bible defines the vision of Project Mars.

If another document conflicts with the Concept Bible:

**The Concept Bible always takes precedence.**

---

# Rule 2 — Document Before You Build

Every major feature must be documented before implementation.

Discussion

↓

Documentation

↓

Approval

↓

Implementation

Never write significant gameplay code without an approved design.

---

# Rule 3 — No Feature Creep

A feature should never be added simply because it sounds exciting.

Before adding any feature, ask:

- Does it support the vision?
- Does it improve gameplay?
- Does it create meaningful choices?
- Does it fit one of the Five Eras?

If the answer is "no," it should not be added.

---

# Rule 4 — Every Feature Needs a Purpose

Every gameplay mechanic should answer at least one question:

- What problem does it solve?
- What experience does it create?
- Why does the player care?

Features without a clear purpose should be removed or redesigned.

---

# Rule 5 — Engineering Is the Core Gameplay

Project Mars is fundamentally an engineering and colony-building game.

Combat may exist, but it must never replace:

- Repairing
- Building
- Exploring
- Manufacturing
- Problem solving

Engineering should remain the primary gameplay loop.

---

# Rule 6 — Respect the Player's Time

Avoid repetitive tasks.

Avoid unnecessary grinding.

Avoid artificial waiting.

Every completed task should provide visible progress.

---

# Rule 7 — Unlock Complexity Gradually

Never overwhelm new players.

Introduce systems through the Five Eras.

Players should naturally grow from survivors into colony builders.

---

# Rule 8 — Real Science, Accessible Gameplay

Project Mars is inspired by real science.

Gameplay should feel believable without becoming a simulation.

The goal is:

Realistic enough to be immersive.

Simple enough to be enjoyable.

---

# Rule 9 — Teach Through Gameplay

The player should learn naturally.

Do not interrupt gameplay with long tutorials.

Instead:

Repairing teaches engineering.

Growing crops teaches biology.

Power systems teach electricity.

Learning should happen through experience.

---

# Rule 10 — Multiplayer Encourages Cooperation

The game should reward teamwork.

Players should naturally adopt roles such as:

- Engineer
- Explorer
- Scientist
- Builder
- Farmer
- Technician

No role should feel unnecessary.

---

# Rule 11 — The Environment Is the Main Challenge

Mars itself is the primary obstacle.

Examples include:

- Dust storms
- Equipment failures
- Radiation
- Frozen pipelines
- Power shortages
- Habitat leaks

Environmental challenges should be more important than enemy encounters.

---

# Rule 12 — AI Guides, Never Plays

The personal AI assistant exists to:

- Explain
- Suggest
- Warn
- Organize

The AI should never solve gameplay automatically.

The player should always make the final decisions.

---

# Rule 13 — Build Modular Systems

Every major gameplay system should be modular.

Examples:

Power System

Robot System

Manufacturing System

Research System

Vehicle System

Each system should evolve independently without requiring major rewrites.

---

# Rule 14 — Prototype Before Expanding

Every new gameplay mechanic should first exist as a small prototype.

Validate:

- Is it fun?
- Is it understandable?
- Does it support the vision?

Only then should it become part of the full game.

---

# Rule 15 — Preserve Existing Work

Avoid rewriting working systems unless there is a clear benefit.

Improve.

Refactor.

Extend.

Do not replace functioning systems without justification.

---

# Rule 16 — Every Major Decision Must Be Recorded

Whenever a significant design decision is made:

Record it in:

PROJECT_DECISIONS.md

Future contributors should understand:

- What changed
- Why it changed
- When it changed

---

# Rule 17 — Ideas Are Not Features

Every new idea belongs first in:

IDEAS_BACKLOG.md

Ideas become features only after review and approval.

---

# Rule 18 — Git Is the Source of Truth

Git history represents the project's development history.

Do not create duplicate files such as:

Concept_Final.md

Concept_Final2.md

Concept_Latest.md

Update the existing document and commit the change.

---

# Rule 19 — Keep Documentation Current

Documentation should evolve alongside the project.

If gameplay changes:

Update the documentation.

Never allow documentation to become outdated.

---

# Rule 20 — Build Something We Would Enjoy Playing

Whenever uncertainty exists, ask:

Would we enjoy this ourselves?

Would this make us feel like pioneers building humanity's future on Mars?

If not, reconsider the design.

---

# Development Principles

Project Mars values:

- Clarity over complexity
- Quality over quantity
- Purpose over novelty
- Engineering over combat
- Exploration over repetition
- Teamwork over competition
- Long-term vision over short-term excitement

---

# Final Reminder

Every feature should reinforce one feeling:

> "We built this."

If players finish a session feeling proud of what they created together, then Project Mars has achieved its goal.
